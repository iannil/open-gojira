# 2026-06-13 每日笔记

## 前端页面接口重构 — 方向确定：TanStack Query

### 背景
12 个前端页面各自手搓 `useState`+`useEffect`+`load()`，loading/error/empty/mutation-refetch 写法 12 种不一致：
- `PlansPage` 无 error 态（只 `message.error`）
- `CandidatesPage` 把 `loading` 挂 Table、吞掉错误
- `DataManagementPage` 用 `refreshKey` 计数器 + `.catch(()=>{})`
- `CockpitPage` 1251 行 monolith（数据编排 + 4 handler + 10 展示组件揉一起）

### 流程
走 `/design-an-interface` 技能，并行派 4 个设计子代理产出** radically different** 方案：
- A. 最小 hook (`useResource`+`PageShell`)，无依赖
- B. 配置驱动框架 (`<ResourcePage spec>`)
- C. TanStack Query + 特性文件夹
- D. Render-prop 组件 (`<Query>`/`<Mutation>`)

对比 5 维度（简洁度/通用性/缓存正确性/深度/误用难度）后，**选定 C**。

### 决策
- 核心分歧是**缓存**：C 把服务端状态缓存当核心抽象（SWR + 去重 + 竞态安全 + devtools），A/D 明确不做。
- 金融控制台对竞态/陈旧数据敏感，库级正确性基线优先于"无新依赖"。
- 借 A 的 feature-folder 拆 Cockpit，借 D 的 `isLoading`/`isFetching` 显式区分（TanStack 原生提供）。
- 否决 B：为 12 页造 react-admin，dashboard 原型打架。

### 落盘
- 完整接口规格 + 6 阶段迁移计划：`docs/progress/2026-06-13-page-interface-tanstack-query.md`
- 项目 ADR：`memory/MEMORY.md` 关键决策 #9

### 待决断（2 点）
1. 默认 `staleTime`：暂定 30s，是否调到 10s？
2. Phase 2（换数据层）与 Phase 3（拆 Cockpit 文件）是否合并？

### 下一步
拍板上述 2 点 → 启动 Phase 0（装包 + Provider + 3 个共享文件，无功能变更）→ Phase 1 试点 PlansPage。

## 决断 (2026-06-13 同日)

1. 默认 `staleTime` = **10s**（金融数据偏短；mutation 都强制失效，10s 主要防反复 mount 重拉；字典类 query 可单独调到 5min）
2. Phase 2 / Phase 3 **保持拆分**（换数据层 + 拆 Cockpit 文件，分两步降风险，各自独立提交）

规格 `docs/progress/2026-06-13-page-interface-tanstack-query.md` 已同步：QueryClient 默认改 10s，"待决断"节改为"已决断"，Next Steps 移除"拍板"项。可直接启动 Phase 0。
