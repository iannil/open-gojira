### 14 项 v2 决策摘要

1. **Q1' schema 统一** — `thesis_variables_json` 现存 3 套不兼容 schema(monitor 期望 `{value, threshold_*}` / sync 写 `{current_value, 无 threshold}` / approve 准备写第 3 套)。标准化为 monitor schema + 修 `sync_stock` 字段名 + 保留 threshold 字段。
2. **Q3' NIM 列** — spec 数据源 shortlist 第一行 `financial:NIM` 映射 `FinancialStatement.net_interest_margin`,但**列不存在**。`financial_pipeline.py` 已经 `_get_nested(m, "nim.t")` 在拉数据,只是没持久化。加列 + pipeline 映射。
3. **Q4' 翻转** — 原 Q4-A "approve 时复制到 thesis_variables_json" 作废。改双源不复制:`research_claim_variables` 表 + `thesis_variables_json` 各为真相源,monitor 双 check 函数。避免两 writer 写同一 JSON 的混乱。
4. **breach_when 机械字段** — 加 `breach_when: "lt" | "gt"` 替代易错的 `direction`。signal 文本 `<1.3%` → breach_when="lt",字面对齐,LLM 不会翻转错。
5. **window_periods 多期检查** — v1 即实现真多期检查(拉过去 N 期连续 breach 才告警)。原"第一版可只支持单点"作废。
6. **业务级 dedup + DB UniqueConstraint** — 同 (stock, variable_name, source) 已有 proposed/active 时 propose skip;DB 加 (research_claim_id, stock_code, variable_name) 唯一约束兜底。
7. **过滤 open holdings** — check_claim_variables INNER JOIN holdings WHERE sell_date IS NULL,卖出即静默,跟 check_held_stocks 对称。
8. **per-var try/except** — 单 var fetch 失败不影响整批,summary 报 checked/breached/failed。
9. **独立 thesis_evaluation_job + last_alerted_at 7 天 dedup** — 不寄生 alert_evaluation_job(scheduler 当前根本不调 check_held_stocks,需新 wiring)。research_claim_variables 加 last_alerted_at,7 天窗口。
10. **告警三处落地** — audit_log (同步) + EventBus ThesisAlertTriggered (新事件) + notification_service.send (复用 NotificationChannel)。
11. **LLM 失败 UX** — propose 成功/失败/部分失败都写 audit_log;Cockpit badge red (失败) / yellow (待 review) 双态。
12. **PATCH endpoint** — `/api/research/claim-variables/{id}` PATCH 改 threshold/breach_when/window_periods,audit_log before/after。原 spec 无 active 态编辑路径,reject 重 propose 又被 dedup 阻塞。
13. **TanStack 30s polling** — 项目无 SSE/WebSocket 基础设施,但已有 TanStack Query (research 页面在用)。Cockpit badge 用 refetchInterval=30s。ClaimVariablesProposed 事件仅 backend 内部 audit 用。
14. **migration 命名 s5_3_** — 原 spec 写 `s5_research_claim_variables.py`,但已有 `s5_1_*` / `s5_2_*`,接链为 `s5_3_*`。

### 实测发现的代码现状

- `FinancialStatement` 列: revenue / revenue_growth / gross_margin / net_margin / npl_ratio / ocf_to_profit_ratio / roe / roa... **没有 net_interest_margin**
- `ValuationSnapshot` 列: pe_ttm / pb / pe_percentile_10y / pb_percentile_10y / dividend_yield
- `thesis_monitor_service.check_variable` 期望 schema `{name, value, threshold_critical, threshold_low, direction}`
- `thesis_variable_sync_service.sync_stock` 写入 schema `{name, current_value, target_condition, unit, source, synced_at}` — **字段名不同 + 无 threshold**
- `scheduler.py` 有 `alert_evaluation_job` (17:30, 跑 alert_service.evaluate_all_rules) 但**不调** check_held_stocks
- `events.py` 7 个 event: DataSyncCompleted / PlanEvaluationCompleted / DraftCreated / AlertTriggered / ResearchRunCompleted / ResearchRunFailed / MonthlyBudgetExceeded — **无 ThesisAlertTriggered**
- `research_claim.stock_codes_json` 是 JSON array,可空可单可多 — schema 单 stock_code 字段已支持每 stock 一行
- 项目无 SSE/WebSocket,有 TanStack Query (research 页面在用)
