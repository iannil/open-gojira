## 四、系统审计发现

### 已知问题（上次审计已记录）

| # | 严重度 | 问题 | 状态 |
|---|--------|------|------|
| 1 | 中 | 筛选器不支持行业过滤 | 未修复 |
| 2 | 低 | 筛选器只返回筛选/排序指标 | 未修复 |
| 3 | 低 | 大量股票 PE/PB 10年分位为 None | 数据限制 |
| 4 | 中 | 财务数据需逐只获取，无批量接口 | 未修复 |

### 新发现问题（本次审计）

| # | 严重度 | 文件 | 问题 | 影响 |
|---|--------|------|------|------|
| **1** | **高** | `screener_service.py` | 批量基本面 API 对大盘蓝筹返回 None，导致筛选器无法捕获已知优质标的 | 筛选器在当前市场环境下几乎无用，必须手动补充候选 |
| **2** | **中** | `valuation_service.py` | `GET /api/valuation/{code}/dashboard` 返回 current_pe/current_pb/current_price 为 None | 综合估值无法自动获取实时价格，需手动传入参数 |
| **3** | **低** | `financial_service.py` | `fetch_and_store_financials` 使用 `db.merge()` 导致重复记录（同一 report_date 出现多条） | 查询最新财务报表时返回重复数据 |

### 问题 #1 详细分析

筛选器的核心流程：
1. `get_company_list()` 获取 5625 只股票代码
2. 每 100 只一批调用 `get_fundamentals()` 获取指标
3. 内存中应用 `_apply_filter()` 过滤

问题出在第 2 步：`get_fundamentals()` 的批量 API 对大盘蓝筹返回的 PE/PB/市值等指标为 None。原因可能是：
- Lixinger 批量 API 的数据覆盖不完整
- 大盘蓝筹的 TTM 指标计算逻辑与个股 API 不同
- 缓存问题导致批量数据过期

建议修复方向：在筛选器中增加 fallback 逻辑，对关键指标为 None 的股票单独调用个股 API 补充数据。

## 五、市场观察

### 当前市场特征（2026年6月）

1. **极度缺乏价值标的**：PE≤20 + 股息率≥3% + 市值≥50亿 仅匹配 5 只（其中 3 只为次新股 SELL 信号）
2. **新股泡沫明显**：筛选器返回的低 PE/低 PB 股票几乎全部为上市不足 3 年的次新股，PB 分位 100%（历史最高）
3. **蓝筹估值分化**：PE 分位偏高但 PB 分位极低（长江电力 PB 0.6%，中国神华 PB 4.8%），说明盈利处于周期高位
4. **买入机会有限**：仅 2 只获强 BUY 信号（中国移动、陕西煤业），4 只获买入信号
5. **银行板块估值回归**：建设银行 PB 分位 63.4%已脱离低估区间，招商银行 PB 40.6%中位

### 投资建议

- **首选**：中国移动（PE 10%/PB 1.2%双低 + 3求 + 75%派息）
- **次选**：陕西煤业（PE 8.8%/PB 6.9%双低 + 18%ROE + 55%派息）
- **防御配置**：长江电力 + 国投电力（水电禀赋不可复制 + 3求 + PB 历史极低）
- **待观察**：中国人寿（27.8% ROE 极具吸引力，但 PE 分位 45% 中位偏高）

## 六、改进建议

### 系统改进

1. **筛选器修复**：解决批量 API 对大盘蓝筹返回 None 的问题（最高优先级）
2. **Dashboard 价格数据**：`/api/valuation/{code}/dashboard` 应返回实时 PE/PB/价格
3. **财务数据去重**：`fetch_and_store_financials` 使用 `merge()` 导致重复，应先查询再插入
4. **筛选器返回全部指标**：不仅返回筛选/排序指标，还应返回 PE/PB/股息率等基本指标
5. **增加"双低"筛选模板**：PE 百分位≤30% 且 PB 百分位≤30% 的一键筛选（排除次新股）

### 流程改进

1. 在筛选器中增加上市年限过滤（排除 3 年以内次新股）
2. 增加"行业龙头"预设模板（每个 SW 行业 Level 1 市值前 3 名）
3. 综合估值支持以 PB 为主（对金融/公用事业/能源行业）
